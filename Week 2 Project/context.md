# Project context (hcde530)

## Who this is for
- **Primary user**: Nina (HCD/UX practitioner) working on HCDE 530 coursework.
- **Secondary audience**: classmates/instructor/portfolio reviewers who may run the examples to understand the approach.
- **Assumed skill level**: beginner-to-intermediate with Python; not a software engineer.

## What success looks like
- **Demonstrate effective data-file processing** (starting from a CSV) in a way that is easy to understand and reuse.
- **Highlight “what matters” in the code**: the important steps, assumptions, and interpretation—without overwhelming readers.
- **Make it runnable**: someone can clone the repo and run a script with minimal setup.
- **Optionally show results in a browser**: provide an output that can be viewed as a “web page” (format to be chosen).

## Current repo structure
- `Week 2 Projects/`
  - `demo_responses.csv`: example dataset (qualitative-ish text responses).
  - `demo_word_count.py`: example script that reads the CSV and prints per-row + summary stats.
- `README.md`: entry point for instructions (expand as needed).

## Data + analysis expectations
- Inputs are small, local files (e.g., CSV) suitable for coursework and demonstration.
- Keep data **non-sensitive** and small enough for GitHub.
- Prefer straightforward, inspectable logic over “clever” optimizations.

## Output expectations (choose one later)
The project may include a browser-viewable output:
- **Static HTML report** generated by a script (easy to share, no server).
- **Jupyter notebook** (explanatory, mixed narrative + code + results).
- **Small local web app** (e.g., Streamlit) for interactive viewing.

Until decided, default to outputs that:
- work on Windows,
- require minimal installs,
- and are easy to explain.

## Engineering constraints + environment
- **OS**: Windows
- **Shell**: PowerShell (avoid `&&` chaining; quote paths with spaces)
- Keep commands copy/paste friendly.

## Code style preferences (for this repo)
- Emphasize clarity and learning:
  - small functions with descriptive names
  - explicit variables for key concepts (participant, role, response, metrics)
  - readable printouts / tables
- Be explicit about assumptions:
  - working directory expectations (relative paths)
  - what counts as a “word”
  - any truncation/formatting in outputs
- Include lightweight guidance:
  - short README steps (“how to run”)
  - examples of expected output

## “Important parts” to highlight (what readers should notice)
- **Reading structured data** reliably (`csv.DictReader`, encoding, headers).
- **Transforming/deriving fields** (e.g., word counts from response text).
- **Validating assumptions** (e.g., required columns exist).
- **Summarizing results** (min/max/average) and presenting them clearly.
- **Reproducible runs** (consistent output, predictable file locations).
